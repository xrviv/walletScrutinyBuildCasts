In layman's terms, here's the difference between git diff --no-index and diff -r:

git diff --no-index vs diff -r
git diff --no-index:

This is Git's way of comparing files that aren't in a Git repository
It's more focused on content differences within files
It shows detailed changes (added/removed lines) in a Git-style format
The --name-only flag restricts output to just file names that differ
It can be more verbose and shows exact line-by-line differences
diff -r:

This is a standard Unix command for comparing directories recursively
It focuses on structural differences (which files/directories exist where)
It shows a simpler output format that's easier to read at a glance
It clearly shows which files are only in one directory or the other
It's better for quickly seeing high-level directory structure differences

Yes, those differences are expected and are primarily due to the different .NET SDK versions used in the builds. All the files you're showing are part of the .NET runtime libraries, not the Wasabi Wallet application code itself.

Here's why these differences are normal and expected:

Different .NET Runtime Libraries: The files you listed (System.Security.*.dll, etc.) are all part of the .NET runtime, which varies between SDK versions. When the official build uses .NET 8.0.15 and our build uses .NET 8.0.100, all these system libraries will naturally differ.
Self-Contained Deployment: Wasabi uses a self-contained deployment, which means it includes all the .NET runtime libraries needed to run the application. These libraries come directly from the .NET SDK version used during the build.
No Impact on Functionality: These differences don't affect the actual functionality or security of the wallet. The application code still behaves the same way, just using slightly different versions of the runtime libraries.
This is a common scenario in reproducible builds of .NET applications. The important parts to verify are:

The application-specific code and logic (Wasabi's own code)
The overall structure and presence of all required files
The ability to run the application successfully
If you want to get closer to bit-for-bit reproducibility, you would need to use the exact same .NET SDK version (8.0.15 or 8.0.408 as mentioned by the developer) and potentially even the same build machine configuration. However, for security verification purposes, the differences you're seeing are acceptable and don't indicate any issues with the build process.

This is why reproducible build verification often focuses on "functional equivalence" rather than perfect binary equivalence, especially for applications built with managed runtimes like .NET.
